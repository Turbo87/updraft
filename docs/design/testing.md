# Testing & Simulation

Testability is an explicit architecture goal. Each layer can be tested and changed on its own. Time enters as data, inputs can be replayed, and the core has a small typed API (see [core.md](core.md)). Tests can therefore run quickly without wall-clock delays. A field bug can become a saved replay test.

## Testing Pyramid

1. **Rust unit tests** (the bulk). Parsers, geodesy, airspace geometry, terrain sampling, and detection heuristics are pure functions. Tests use `insta` snapshots against a `testdata/` corpus of recorded captures from real devices, a deliberately cultivated shared asset. **Property-based fuzzing via `proptest`:** Bluetooth/TCP input is untrusted, so every parser crate — `updraft_nmea` with each of its vendor families (FLARM, LXNav, OpenVario, Cambridge), plus OpenAir, CUP, IGC — carries proptest suites asserting no-panic on arbitrary input plus round-trip properties where applicable. These run in normal CI.
2. **Core scenario tests.** The deterministic state machine pays off: feed a scripted input sequence (fixes, sentences, commands, injected timestamps) into `handle()` and assert on the returned changes, effects, and snapshots — effects-as-data means the warning→audio path is assertable with no mocks. Whole-flight scenarios (takeoff detection, airspace warning sequence, device loss and reconnect) run in milliseconds at simulated time, as a plain loop with no async runtime.
3. **Runtime invariant tests.** The shared runtime carries the stream contract and must pin it: atomic subscribe (a late subscriber's snapshot already contains earlier submissions), FIFO change ordering, slow-subscriber drop-and-reconnect, block-on-full input backpressure, and worker-panic-to-`JobFailed` conversion.
4. **Contract tests.** Every command/change schema round-trips Rust ↔ generated TypeScript. CI fails if generated types drift from committed ones.
5. **Playwright e2e, the flagship layer.** Tests launch `updraft-server` with a **simulation profile**: replay inputs from an IGC/NMEA file at N× speed, fixed datasets, injected timestamps (the simulation seam accepts `observed_at`, so e2e time is genuinely simulated, not wall time). Playwright drives the real frontend in a real browser: _"load this flight, tap the map here, expect the airspace dialog to list TMA Langen."_ No Tauri, no device, no real-time flakiness. Visual regression on key screens optional later. Since the Tauri app embeds the same axum server the Playwright suite targets, these tests exercise the shipping frontend, core, runtime, and transport — only the native plugins (and, in the hybrid asset shape, asset serving) remain outside their reach — without requiring native automation on five platforms. The e2e suite runs as its own CI job with a cached/prebuilt server binary. **WebGL in CI:** stock headless Playwright Chromium renders MapLibre via its built-in software GL with no special setup. Map assertions target map state (`getStyle`, `getLayer`, `queryRenderedFeatures`) rather than pixels, so they hold across GL rasterizers. Screenshot comparison stays optional polish. The frontend exposes a `testMode` flag that disables map easing/animations and other nondeterminism sources, and tests await explicit "map idle" / "data version N rendered" signals rather than timeouts.
6. **Thin Tauri smoke tests.** The shell is kept so thin that a small manual/emulator checklist (plugin permissions, foreground-service lifecycle, embedded-server lifecycle across suspend/resume) suffices. Everything above it is already covered.

A **CI verify mode** also replays recorded flights while workers calculate their results again (see [replay.md](replay.md#ci-recompute-verification)). It compares result hashes and reports the first different input and worker. This is easier to debug than only seeing a different final state.

## Simulation as a Feature, Not Scaffolding

The same replay machinery ships to users: IGC replay mode and a simple simulator (set position/speed/track, fly with buttons or by dragging the ownship symbol) for ground training and demos. Dragging conflicts with the normal tap-to-query and pan gestures, so simulator mode uses a **distinct interaction**, a dedicated drag handle on the ownship symbol (or an explicit "move aircraft" mode) rather than overloading the plain drag gesture. The TCP transport doubles as the Condor interface _and_ a test injection point.

## CI

GitHub Actions: Rust tests + clippy, frontend check/lint, Playwright suite, contract-drift check, release builds per platform. The workflows themselves are audited by [zizmor](https://docs.zizmor.sh/).
